How the pieces fit together
🧩 Skills
Reusable know-how Claude applies automatically when your request matches — or on demand via /skill-name.
🤖 Agents
Specialists Claude delegates to — each with its own instructions and tool limits, for one job done well.
🔗 Hooks
Scripts that fire automatically at lifecycle events — block dangerous commands, auto-run tests, log everything.
⌨️ Commands
Your own /slash-commands — a typed shortcut that runs a whole scripted procedure.
🔌 Plugins & MCP
External powers: tools Claude can call (GitHub, databases, browsers…) and installable plugin bundles.
🔄 Workflows
All of the above working as one system — installed together, runnable end-to-end with a single command.

Controls what Claude may do without asking. Allow rules skip the permission prompt; deny rules block the action outright (deny always beats allow).
💡 Why use it: More autonomy where it’s safe, hard limits where it isn’t — stop being asked to approve git status for the hundredth time, while making .env untouchable.
Example: Bash(git *) in allow lets every git command run unprompted. Read(.env) + Read(**/secrets/**) in deny means Claude can never read your secrets, even if asked. Rule syntax: Tool(specifier) — e.g. Bash(npm run test:*), WebFetch(domain:github.com), Edit(~/.ssh/**).

Scripts in ~/.claude/hooks/. Files here do nothing until wired to an event in the ⚡ Events tab.
Named, derived events — a detector script bound to a built-in event that fires on one precise condition.
What is a custom event?
Claude Code only fires its built-in events — a custom event is a derived one: a hook bound to the right built-in event whose script detects one precise condition and acts only then, giving that condition a name of its own.
💡 Why use it: "when Claude pushes to git", "when a file over 1 MB is written", "when tests fail" aren't events Claude Code knows — but a custom event makes them real, reusable triggers.
Example: GitPushDetected = a PreToolUse hook matching Bash that fires only on git push commands and blocks them until tests have run.

Keybindings
Custom keyboard shortcuts (keybindings.json)
Customizes keyboard shortcuts in the Claude Code terminal UI via ~/.claude/keybindings.json (requires Claude Code ≥ 2.1.18; run /keybindings in Claude to open the same file). Changes apply instantly — no restart.
💡 Why use it: rebind actions to keys you actually reach for, get vim-style navigation, or unbind defaults that clash with tmux/screen. Every binding lives in a context (Chat, Confirmation, Transcript…) so the same key can do different things in different places.
Example: bind ctrl+e → chat:externalEditor to draft prompts in your $EDITOR, or set ctrl+b → null to unbind it because tmux owns it. Chords work too: ctrl+k ctrl+s.

📚 Reference — contexts, keystroke syntax, reserved keys
Keystroke syntax: modifiers joined with + (ctrl, shift, alt/meta, cmd); chords separated by spaces (ctrl+k ctrl+s); special keys escape enter tab space up down left right backspace delete; a bare uppercase letter implies Shift.
Unbinding: map a key to null to remove a default.
Reserved (cannot rebind): Ctrl+C (interrupt), Ctrl+D (exit), Ctrl+M (= Enter).
Terminal conflicts: Ctrl+B (tmux prefix), Ctrl+A (screen), Ctrl+Z (suspend).
Contexts: Global · Chat · Autocomplete · Settings · Confirmation · Tabs · Help · Transcript · HistorySearch · Task · ThemePicker · Attachments · Footer · MessageSelector · DiffDialog · ModelPicker · Select · Plugin · Scroll — a binding only fires while its context is active. Full docs: code.claude.com/docs/en/keybindings
